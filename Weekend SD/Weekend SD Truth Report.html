3 · Drift over time (is a frozen constant defensible?)

The lines are the trailing 3-month (13-weekend) rolling mean and SD across the whole history; the dashed lines are Igor's frozen constants. If the rolling line wandered far from the dashed line, a single frozen number cannot represent the live distribution.

Rolling 3-month mean ranged 1.04%–6.75% for UP (frozen 2.28%) and 1.05%–6.82% for DOWN (frozen 2.37%). The distribution is non-stationary: the rolling mean swings by several multiples of the frozen value, so live 1-month / 3-month windows are meaningfully different from any single frozen constant.

5 · Where did the frozen numbers come from?

We searched every trailing window ending at the last in-sample weekend and every calendar year for the sub-period whose statistics best fit the six frozen constants (relative-L2 distance). Best match: trailing_104wk_end2024 (n=104), and even it still differs from the frozen constants by about 15% on average per number. That window's stats were UP mean 2.440 / SD 2.569 / p95 6.666, DOWN mean 2.041 / SD 2.189 / p95 5.464. So no in-sample sub-period cleanly reproduces the frozen numbers. The closest is the last two years (2023–2024), which is directionally consistent with the constants having been computed over a recent, calmer lookback — most likely one ending in 2025–2026 data that we deliberately do not hold in this in-sample study, or under a slightly different definition.

6 · What this means for the indicator

  • The frozen constants are a snapshot, not the full-history truth. Use the corrected full-history numbers in Section 1 as the static baseline.
  • Because the rolling mean and SD drift substantially (Section 3), the planned live 1-month / 3-month windows are the right design — a single frozen number systematically mis-sizes the Monday levels whenever volatility regime shifts.
  • UP and DOWN are close but not identical; keep them separate as the indicator already does.
  • DST is handled per-zone, so the Friday-NY start and Monday-Dubai base survive the March/November New York clock changes without drifting an hour.

Methodology

  • Definitions reproduced exactly from Weekend_Monday_SD_Levels_V2.pine: weekend = Friday 17:00 America/New_York → Monday 02:00 Asia/Dubai; base = close of the last 1-minute bar strictly before Monday 02:00 Dubai; Monday window = Monday 02:00 → Tuesday 00:00 Dubai; UP = (Monday high − base)/base, DOWN = (base − Monday low)/base.
  • Timezones via Python zoneinfo for both zones. Dubai is a fixed UTC+4 (no DST); New York DST is applied automatically to the Friday 17:00 anchor. No fixed offsets are used anywhere.
  • Data: btcusdt_krown_1m_perp_full.parquet, 2,795,040 one-minute bars, 2019-09-09 00:00 UTC → 2024-12-31 23:59 UTC. Timestamps are UTC epoch open-time (Jesse convention). The file is a continuous 1-minute grid with zero missing minutes and zero duplicates. In-sample only: nothing after 2024-12-31 23:59 UTC is loaded.
  • Dropped weekends: base-bar gap 0, Monday-window gap (>10% missing) 0, edge/out-of-coverage 1 (partial windows at the very start/end of the dataset, not a data-quality fault). Zero-volume Monday windows kept but flagged: 0.
  • Statistics: mean and standard deviation via numpy; sample SD uses ddof=1 (also reporting ddof=0). p95 via numpy.percentile default (linear interpolation).
  • Determinism: pure function of the candles, stable mergesort, no randomness. Two consecutive runs produce a byte-identical weekend_sd_stats.csv (md5 verified).
